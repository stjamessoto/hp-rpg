# HP-RPG

A text-based Harry Potter RPG: canon-faithful character creation, a
Sorting Hat and Ollivander's flow, and a data-driven playthrough of all
seven years at Hogwarts, ending at either an ordinary graduation or the
Battle of Hogwarts depending on which year the character starts in. Built
as v1 for the browser, with a portable `/core` designed from day one to be
dropped into a React Native mobile shell without edits.

**The single most important constraint is canon fidelity.** Every fact the
game states is traceable to a source and tagged with a confidence tier; see
[Canon data & the `canonTier` strategy](#canon-data--the-canontier-strategy)
below.

## Setup

Requires Node 18+ (uses the global `fetch`/`AbortController`).

```bash
npm install
npm run ingest:offline   # or `npm run ingest` to also attempt live wiki enrichment — see below
npm run dev               # http://localhost:5173
```

Other scripts:

```bash
npm run build              # typecheck (core + web + scripts) then vite build
npm run test                # vitest run (core/tests/**)
npm run typecheck           # tsc --noEmit across core, web, and scripts
```

`data/*.json` is generated output, not hand-edited — to change a fact, edit
the corresponding file under `/scripts/seed` and re-run `npm run ingest`
(or `ingest:offline`).

## Canon data & the `canonTier` strategy

Every record in `/data/*.json` — and every nested sub-fact (a head of
house, a professor's tenure, an era note) — carries two fields:

- **`canonTier`**: `"book"` | `"supplementary"` | `"wiki-only"`
- **`source`**: a human-readable citation (a book + chapter, or a URL)

`canon/style-bible.md` and `core/src/canon/validator.ts` are both written
to treat these tiers as a trust ladder: **book beats supplementary beats
wiki-only**, and nothing is ever invented to fill a gap — unknown fields
are `null`.

Ingestion (`scripts/ingest-canon.ts`) works in that same priority order:

1. **The books.** Facts well-established in the seven novels are encoded
   directly in `/scripts/seed/*.seed.ts` as `"book"` tier, each with a
   chapter-level citation. This is ground truth and is never overridden.
2. **The Harry Potter Wiki**, queried live via its MediaWiki API
   (`harrypotter.fandom.com/api.php`), used *only* to fill fields the seed
   data left `null` (e.g. a birth year with no confirmed book source).
   Anything sourced this way is tagged `"wiki-only"` and the script prints
   a warning to manually review it before it's ever promoted to a higher
   tier. The network call has a 6s timeout and fails soft — no network
   (`npm run ingest:offline`, or just no connectivity) still produces the
   full `"book"`/`"supplementary"` dataset.
3. **Wizarding World / Pottermore** author writing that doesn't contradict
   the books — also hand-encoded in the seed files, tagged
   `"supplementary"` (e.g. exact birth dates for characters the novels
   don't date precisely, or the Wand Woods essays).

This is a curated, high-confidence subset of canon (four houses, ~25
spells, ~25 characters, a 1926–2017 timeline anchored to the core
1991–1998 years, key locations, ~15 creatures, all core/elective subjects
with their famously rotating Defence Against the Dark Arts professors, and
the four blood-status categories) — not an exhaustive wiki mirror. Extend
it by adding entries to `/scripts/seed` with correct `canonTier`/`source`,
never by editing `/data/*.json` directly.

## Canon validator

`core/src/canon/validator.ts` checks a `Character` (or a proposed in-fiction
fact) against the data layer:

- `validateAgeViolation` — is the character's age plausible for the chosen
  Hogwarts start year?
- `validateHouseLock` — does the Sorted house contradict a backstory tag
  that pinned the house in advance?
- `validateBackstoryCanon` — does free-text backstory claim something a
  closed canon family composition rules out (e.g. "the Potters' second
  child")?
- `validateSpellEra` — is a spell being used before its first canonical
  appearance?
- `validateNpcPresence` — is a canonical NPC referenced before their birth
  year or after their death year?

`validateCharacter` runs the character-scoped checks together;
`createCharacterFlow` calls it automatically at the end of character
creation and prints any violations. Unit tests: `core/tests/validator.test.ts`.

## Character creation

`core/src/character/createCharacter.ts` drives the full flow purely
through the `UiAdapter`/`RngAdapter` interfaces — identity → blood status →
backstory → appearance → Sorting Hat → Ollivander's wand → attributes —
and is exercised end-to-end in `core/tests/createCharacter.test.ts` via a
scripted `ScriptedUiAdapter` and a seeded RNG (proving both the flow and
its reproducibility). The result is a storage-agnostic JSON `Character`
(`schemaVersion: 1`) saved through `StorageAdapter` — `localStorage` on
web, device storage on mobile.

**Choosing your house.** `core/src/character/sortingHat.ts` offers two
paths, asked as the very first sorting question: let the Hat decide (the
canon-faithful default — a short quiz, then the Hat can additionally
"consider" a stated preference the way it does Harry's "not Slytherin,"
without guaranteeing it), or tell it directly which of the four houses you
want. Picking a house directly is honest about what it is — a deliberate
player-agency option layered on top of the books' actual mechanism, not a
claim that the Hat works this way in canon.

## Save files

Two things get saved independently, both through `StorageAdapter` (so the
same code works against `localStorage` or device storage):

- **The character** (`core/src/character/save.ts`) — identity, house,
  wand, backstory, everything from character creation. Multiple characters
  can exist at once; `listCharacterIds` enumerates them.
- **Game progress** (`core/src/game/gameSave.ts`) — which of the 104 scene
  checkpoints a given character is currently on. `runGameLoop` saves this
  automatically after every scene transition when a `StorageAdapter` is
  passed in via its `options.storage`, and clears it the moment a
  playthrough reaches a natural end (so a finished story doesn't show up
  asking to be "continued").

Both `web/src/main.ts` and `mobile/App.tsx` check for existing saves on
launch: if any exist, you get a menu — "Continue as `<name>` (`<house>`,
Year `<N>`)" for each in-progress character, "`<name>` — story complete"
for a finished one, plus "Start a new character." Continuing resumes by
replaying the exact scene you were last looking at (not fast-forwarding
past it), so you're never left guessing whether your last choice actually
registered. Verified for real in both a browser and Expo's web target:
create a character, play a few scenes, reload the page (simulating
closing and reopening the app) — the continue menu appears with the right
house and year, and picking it resumes exactly where you left off.

## Style bible

`canon/style-bible.md` is the single reference for how any prose in this
game should read — voice, POV, British diction, a do-not-use list, and
tone-by-era guidance matching the books' own escalation. `/data/scenes/*`
prose was written against it. It ends with a ready-to-paste **prompt
-injection block** (§6) meant to be prepended to any future LLM call that
generates in-fiction text, plus in-voice examples and labeled violation
counter-examples, so both a human author and an LLM can be checked against
the same standard.

## Gameplay loop: all seven years

`core/src/game/sceneEngine.ts` renders `Scene` records
(`/data/scenes/*.json`, 104 of them) and resolves player choices;
`scenesLoader.ts` assembles them into one continuous chain spanning the
character's entire time at Hogwarts — starting in **Diagon Alley** (school
shopping, Gringotts, Madam Malkin's) and the **Hogwarts Express** (the
trolley witch, the countryside rolling past) before year 1 proper begins,
through year 7. Each `Scene` declares its own `hogwartsYear` (1-7); the
engine derives the real calendar year from
`character.identity.hogwartsStartYear + (hogwartsYear - 1)`, so the same
scene content adapts to *when* a given character was at school.

104 scenes is roughly 3x the game's original 35-scene backbone, added
specifically to give every year real breadth rather than just a single
"canon event, then move on" thread. Alongside the set-piece arcs below,
every year now has its own Quidditch progression (tryouts, reserve
matches, a captaincy by year 6), more class and elective variety
(Astronomy, History of Magic, Ancient Runes, N.E.W.T. electives), and
holiday/downtime beats (Christmas at the castle, Hogsmeade trips,
prefect duties, house-elf welfare, friendships tested and kept) that
exist independent of *which* calendar year a character's cohort landed
in.

**Every decision point also accepts free text.** Alongside a scene's
written choices, `runScene` always adds a "Something else — type your own
action" option; picking it asks what your character does, in your own
words, and narrates a result (via the optional Gemini narrator if
configured, or a plain "You `<what you typed>`." acknowledgement if not).
The 104 checkpoint scenes stay fixed regardless — think of them as a
guideline of *where the story is* (which year, which canonical events are
happening around you) rather than a strict script of *what you do at each
one*. This works because, structurally, a scene's written choices already
all funnel to the same next checkpoint (the only real forks are the two
automatic `branch`es below, which depend on the character, not on which
option was clicked) — so resolving a typed action through the scene's
first choice is always safe, including at the one scene that *is* a
branch. See `core/src/game/narratorPrompt.ts`'s `buildFreeActionContext`
for how the prompt explicitly tells the model to treat the player's text
as an in-fiction action to narrate, never as an instruction to itself,
even if it's phrased like one.

A few things make a 7-year, canon-faithful loop tractable without writing
per-start-year variants of everything:

- **The `{{wider-world}}` token.** Most years include a "word from beyond
  the castle walls" scene whose prose ends in this token.
  `core/src/game/widerWorld.ts` resolves it by looking up that year's
  canon events straight from `timeline.json` (a hand-placed table maps
  each school year to the right event ids, since events are dated by real
  calendar year but a school year spans two of them — see the code comment
  for why e.g. the Battle of Hogwarts, 2 May 1998, belongs to school-year
  1997, not 1998). Years with no recorded events get a graceful "it was a
  quiet year" fallback rather than an invented event. The same character
  starting in 1991 will see the Triwizard Tournament and Voldemort's
  return in their *own* year 4; a character starting in 1993 sees it in
  year 2; a character starting in 1995 never sees it at all, because it
  had already happened before they arrived. None of this is hardcoded per
  start year — it all falls out of the one lookup table plus arithmetic.
- **The `{{dada-professor}}` token**, resolved the same way from
  `subjects.json`'s `professorsByEra`, so class-flavor scenes always name
  the right professor for the famously-cursed Defence Against the Dark
  Arts post in whatever year the character is actually in.
- **The year-7 war/peace fork.** Every playable start year's own year 7
  lands on a different calendar year (1997 through 2003), and exactly one
  of those — 1997 — is Deathly Hallows' school year: Hogwarts under
  Snape's headmastership and the Carrows, danger for Muggle-born students,
  ending at the Battle of Hogwarts. Every other calendar year is an
  ordinary, peaceful N.E.W.T. year ending in graduation.
  `end-of-year-six.json` encodes this as a `branch` on its single choice
  (`on: "calendarYear"`, `targetHogwartsYear: 7, calendarYearAtLeast:
  1998`) rather than two copies of six years of content —
  `sceneEngine.ts`'s `resolveBranch` evaluates it once, at that one
  decision point.
- **Calendar-gated set-piece arcs, attached at one "canonical" slot.**
  Five years (2, 3, 4, 5, 6) each carry a detailed, multi-scene arc for a
  specific real-book event — the Chamber of Secrets (1992), Sirius Black
  and the Dementors (1993), the Triwizard Tournament (1994), Umbridge and
  Dumbledore's Army (1995), Slughorn and Dumbledore's death (1996). Writing
  a full version of each arc for every possible calendar year a character
  might land in that `hogwartsYear` would mean dozens of near-duplicate
  scene chains. Instead, a new branch kind — `on: "calendarYearLookup"`
  (`{ targetHogwartsYear, cases: { [calendarYear]: sceneId }, fallback:
  sceneId }`) — routes a character into the detailed arc only when their
  *own* calendar year at that `hogwartsYear` matches, and into a shorter
  "ordinary year" scene (`ordinary-second-year-y2.json`, etc.) plus the
  `{{wider-world}}` digest otherwise. The detailed arcs are deliberately
  scoped to the single calendar year that would make a character's cohort
  match Harry's own — i.e. a character who started Hogwarts in 1991. A
  character starting in a different year still gets correct coverage of
  the same real event (via `{{wider-world}}`, if it falls somewhere in
  their seven years at all) — just without a dedicated multi-scene arc
  built around it. This keeps the content investment concentrated on the
  one cohort every canon reader actually recognizes, rather than thinning
  it across every mathematically possible start year.

The player character is always an original student, never a
stand-in for Harry: they live their own school life (classes, an elective
pick, Hogsmeade, O.W.L.s/N.E.W.T.s) while major canonical events happen
around them as the `{{wider-world}}` digest and — for a couple of
deliberately chosen pivotal beats, like joining the reconstituted
Dumbledore's Army in year 5 or fighting at the Battle of Hogwarts as one of
many unnamed defenders — direct but background participation, never
rewriting what a named canonical character does. `core/tests/sceneEngine.test.ts`
plays a full scripted 7-year run down both branches (an 1991 start into
the wartime ending, a 1993 start into graduation) as a regression test
that the whole chain still resolves correctly end to end.

Adding a scene is still: write the JSON, add one import line to
`scenesLoader.ts` — no engine changes required, unless it needs a new
branch point.

**Backstory and blood status change what happens, not just what's
recorded.** Two concrete mechanisms, both deterministic (no LLM involved —
see the note on an AI-GM layer under Phase 3 below for that alternative):
a `{{upbringing-reaction}}` token (`core/src/game/reactions.ts`) gives the
Sorting Feast a different first reaction depending on
`backstory.upbringing` (wizarding / Muggle / mixed households read that
first night differently), and `SceneChoice.branch` now supports branching
on `bloodStatus` as well as calendar year — a Muggle-born character on the
wartime year-7 branch is routed through an extra scene
(`muggle-born-in-hiding.json`, the Muggle-born Registration Commission)
that a pure-blood or half-blood character never sees. Both are covered by
`core/tests/sceneEngine.test.ts`.

## Optional: LLM-backed scene enrichment

Two touchpoints, both entirely optional and off by default — the game is
fully playable, and was built and tested, with the narrator disabled:

1. **A personalized beat on every scene** — a short paragraph reacting
   specifically to *this* character's backstory, blood status, appearance,
   or wand, on top of (never instead of) the hand-written prose. Fires
   unconditionally once per scene.
2. **Narrating typed custom actions** — see "every decision point also
   accepts free text" above. When a narrator is configured, picking
   "type your own action" gets a real in-voice response to what you
   typed; without one, a plain acknowledgement.

- **`core/src/adapters/NarratorAdapter.ts`** defines the interface, kept to
  the exact same injectable-adapter pattern as `StorageAdapter`/
  `RngAdapter`/`UiAdapter`: `/core` never imports a vendor SDK or makes a
  network call itself. `NullNarratorAdapter` (the default `runGameLoop`
  falls back to when no narrator is passed) always returns `null` — zero
  cost, zero behavior change.
- **`core/src/game/narratorPrompt.ts`** builds both prompts, pure and
  network-free. `buildNarratorContext` (the personal beat) fills in
  canon/style-bible.md §6's injection block with the current story
  year/house, and assembles a `groundingFacts` object from *only* this
  character's own state (backstory, blood status, appearance, wand) —
  grounding is deliberately narrow so the model has little room to invent
  beyond the specific character detail it's meant to react to.
  `buildFreeActionContext` (the typed-action narration) shares the same
  system prompt but quotes the player's raw text back with explicit
  instructions to treat it strictly as an in-fiction action to narrate,
  never as a command to the model — a lightweight defense against prompt
  injection via what the player types.
- **`web/src/adapters/GeminiNarratorAdapter.ts`** and
  **`mobile/src/adapters/GeminiNarratorAdapter.ts`** are the two pieces
  that actually talk to a network — byte-for-byte identical files, since
  it's a plain `fetch` call to Gemini's `generateContent` REST endpoint
  with no SDK dependency, no browser-specific API. Any other model/vendor
  works the same way on either platform — implement
  `NarratorAdapter.generate()`, swap it in.
- **Model note**: defaults to `gemini-flash-latest`, an auto-updating
  alias (currently resolves to `gemini-3.5-flash`) — an earlier pin to
  `gemini-2.5-flash` 404'd because Google retired it for new API keys.
  Also disables Gemini 3's "thinking" mode (`thinkingConfig: {
  thinkingBudget: 0 }`): for a one-paragraph reaction it burned ~15x more
  tokens on reasoning than on the actual output, with no quality
  difference in testing.

**To enable it**:
- Web: `cp web/.env.example web/.env`, fill in `VITE_GEMINI_API_KEY`.
- Mobile: `cp mobile/.env.example mobile/.env`, fill in
  `EXPO_PUBLIC_GEMINI_API_KEY` (the `EXPO_PUBLIC_` prefix is required —
  that's what tells Expo's Metro config to inline it into the app bundle).

Get a key at [aistudio.google.com/apikey](https://aistudio.google.com/apikey).
Both `.env` files are gitignored — a key never gets committed. Leave
either unset and nothing changes; `main.ts`/`App.tsx` only construct a
`GeminiNarratorAdapter` if their respective env var is present. Verified
working end-to-end on both platforms with a real key: real Gemini calls
correctly grounded generated prose in the specific character's wand,
backstory, and appearance with no invented details.

**Security note**: calling a generative-AI API directly from client code
means the API key ships in every request the app makes, visible to anyone
who opens dev tools (web) or inspects network traffic (mobile). That's
fine for local, personal use (the intended use here) but is **not** safe
for a publicly distributed build — a public release should proxy this
call through a small backend that holds the key server-side instead.

## Mobile migration roadmap

### Phase 1 — this build

`/core`, `/data`, and `/canon` are portable as-is: `/core` has no `window`,
`document`, or Node-only import anywhere (see the dependency-audit note
below), all I/O goes through the three injectable adapters
(`StorageAdapter`, `RngAdapter`, `UiAdapter`), and `/data`/`/canon` are
plain JSON/Markdown. `/web` is the only web-only layer — it supplies
`LocalStorageAdapter` and `DomUiAdapter`, both trivial, single-purpose
files that a mobile shell replaces wholesale rather than reuses.

### Phase 2 — React Native (Expo) shell — built

`/mobile` is a working Expo (managed workflow) app that imports `/core`
unchanged and supplies the same three adapters `/web` does, each a
drop-in replacement for its browser counterpart:

| Interface | Web implementation | Mobile implementation |
|---|---|---|
| `StorageAdapter` | `LocalStorageAdapter` (`window.localStorage`) | `mobile/src/adapters/AsyncStorageAdapter.ts`, backed by `@react-native-async-storage/async-storage` — same three methods, no core changes |
| `RngAdapter` | `createSeededRngAdapter` (already engine-neutral) | the same file, imported as-is — mulberry32 is pure arithmetic, no platform dependency at all |
| `UiAdapter` | `DomUiAdapter` (DOM log + form/buttons) | `mobile/src/adapters/NativeUiAdapter.ts` — a small observable store (`print`/`ask`/`choose`/`confirm` update `state` and notify subscribers) that `App.tsx` subscribes to via `useSyncExternalStore`, rendering a `ScrollView` log plus a `TextInput` or button list depending on what `/core` is currently asking for |

Run it:

```bash
cd mobile
npm install
npx expo start
```

Then scan the QR code with the **Expo Go** app on your phone (iOS or
Android) — no app-store submission, no native build step, running in
under a minute. `npm run typecheck` (in `/mobile`) and
`npx expo export --platform android` (a one-shot bundle-only check, no
device needed) both verify the app compiles and bundles cleanly.

**Bundler config, and a Windows-specific gotcha worth knowing about:**
`metro.config.js` aliases `@core`/`@data` to the real `/core/src` and
`/data` directories (mirroring Vite's `resolve.alias` and the root
`tsconfig.base.json`'s `paths`), and includes a small custom resolver so
Metro follows `/core`'s `.js`-suffixed relative imports (a TS
"Bundler"-moduleResolution convention Vite/tsc resolve natively but Metro
doesn't). On this Windows dev machine, with no Watchman installed, Metro's
fallback JS file-crawler failed to register files reached only via
`watchFolders` pointing *outside* the Expo project root ("Failed to get
the SHA-1 for ..." for a file that demonstrably exists) — a known rough
edge for Metro on Windows in monorepo-style layouts. The fix in
`metro.config.js`: two NTFS directory junctions, `mobile/_core` → `../core`
and `mobile/_data` → `../data` (created once via `mklink /J`, gitignored,
regenerate with the commands in `metro.config.js`'s comment if you ever
delete them), so the real content is reachable from *inside* Metro's own
project root and the default single-root crawl covers it with no
special-casing. If you're on macOS/Linux, or have Watchman installed, you
may not need this workaround at all — try the plain `watchFolders`
approach first (see git history / the comment in `metro.config.js`) before
resorting to junctions.

### Phase 3 — mobile UX layer — partially built

The optional AI-GM integration is done: `mobile/src/adapters/GeminiNarratorAdapter.ts`
is a byte-for-byte copy of `/web`'s (plain `fetch`, no browser API, so it
needed zero changes), wired into `App.tsx` reading the key from
`process.env.EXPO_PUBLIC_GEMINI_API_KEY` (Expo's Metro config inlines
`EXPO_PUBLIC_*` vars from `.env`, the same idea as Vite's `VITE_*`/
`import.meta.env` — see `mobile/.env.example`). Verified end-to-end via
Expo's web target (`react-native-web`, so the exact same `App.tsx`/adapter
code is browser-testable): a live Gemini call correctly wove a character's
wand, backstory, and appearance into an in-voice paragraph.

Still to build on top of the now-working Phase 2 shell: `/mobile`'s
`App.tsx` today is deliberately the same single-screen shape as `/web` —
one scrolling log, one control strip below it. A proper multi-screen app
(a character-sheet screen reading directly from the saved `Character`
JSON, a save-slot manager — `listCharacterIds`/`loadCharacter`/
`deleteCharacter` in `core/src/character/save.ts` are already
slot-agnostic, nothing in `/core` needs to change for this) and nicer
choice-button styling than the current plain list are still open.

### Dependency-audit note

`/core` was checked for anything that would break under React Native:

- No `window`, `document`, `navigator`, or other DOM globals anywhere in
  `core/src`.
- No Node built-ins (`fs`, `path`, `crypto`, etc.) — the seeded RNG
  (`createSeededRngAdapter`) is hand-rolled arithmetic specifically so it
  doesn't need Node's `crypto` or the browser's `window.crypto`; the one
  `Date` usage (`character.createdAt`) is a standard JS global available
  identically in Node, browsers, and Hermes/JSC (React Native's JS
  engines).
- All JSON imports go through the `@data` alias, not `fetch`/`fs`, so they
  bundle as static assets on every target.
- The only files that *do* touch `window`/`document`
  (`web/src/adapters/LocalStorageAdapter.ts`,
  `web/src/adapters/DomUiAdapter.ts`) live entirely under `/web` and are
  exactly the two files Phase 2 replaces.

`npm run typecheck:core` enforces this at the type level (`web/DOM`
lib types aren't in `core/tsconfig.json`'s `lib` array, so any accidental
`window`/`document` reference in `/core` fails to compile).

## Desktop app (Electron)

`/desktop` wraps `/web`'s existing build in a native window — same site,
same `/core`, zero changes to either. `desktop/main.cjs` is the entire
Electron main process: in dev mode it loads the live Vite dev server (the
same hot-reloading loop as the browser, just in a window); in production
mode it loads a built copy of the site from `desktop/www/` (populated by
`prepare-www.cjs`, which copies the root `../dist` in — electron-builder
only packages files from inside the app directory, not a parent one, so
the built site has to live inside `/desktop` before packaging). That
needed one small change elsewhere: `vite.config.ts` now sets `base: "./"`
so built asset paths are relative (`./assets/...`) rather than
root-absolute, since there's no server to resolve an absolute path against
when loading over `file://`. Harmless for normal browser/static hosting.

### Dev mode

```bash
cd desktop
npm install
npm run dev      # hot-reloading dev window
```

From the repo root, `npm run desktop:dev` does the same without an extra `cd`.

### The actual installed app

This is genuinely packaged now, not just a dev-mode window — a portable
`HP-RPG.exe` (via `electron-builder`) with a Desktop shortcut, both
already set up:

```bash
cd hp-rpg
npm run build           # builds /web into /dist
cd desktop
npm run dist             # packages /dist into desktop/release/HP-RPG.exe
```

`desktop/release/HP-RPG.exe` is fully self-contained (~90MB, bundles its
own Chromium/Node runtime, as every Electron app does) — no installer, no
"Programs and Features" entry, just a file you can copy/move/delete
freely. **A shortcut to it (`HP-RPG.lnk`, custom icon) already exists on
your actual Desktop folder** (`[Environment]::GetFolderPath("Desktop")`
resolved it to your OneDrive-redirected Desktop, not the plain
`C:\Users\<you>\Desktop` — worth knowing if you ever go looking for the
`.lnk` file by hand). Double-click it to play. If you ever rebuild the
`.exe` (say, after changing a scene), the shortcut still points at the
same path, so no need to recreate it.

### The icon

`desktop/build/icon.svg` — an original wand-and-spark emblem in the app's
existing gold-on-dark palette, not the official Harry Potter/Wizarding
World logo or any copyrighted artwork (those are trademarked; this is
deliberately its own thing, just in the same spirit). Rasterized to
`icon.png` (512×512, used by `/web`'s favicon at a smaller size too) and
`icon.ico` (a proper multi-resolution Windows icon: 16/32/48/64/128/256px
baked into one file, so it stays crisp from a taskbar-sized icon up to a
large-icons desktop view) via `sharp` + `png-to-ico`. Neither is a
permanent project dependency — installed with `--no-save`, used once,
removed; regenerate by reinstalling them temporarily and re-running the
same resize/pack steps against `icon.svg` if you ever want to tweak the
design.
